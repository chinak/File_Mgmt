# 档案管理系统 - 需求优化文档

> 文档版本：v1.0  
> 编写日期：2026-03-23  
> 文档性质：业务逻辑优化需求

---

## 一、文档概述

本文档基于对档案管理系统现有业务逻辑的全面分析，梳理出当前实现中存在的业务逻辑不完善之处，并提出相应的优化建议。文档聚焦于业务流程合理性、数据完整性和用户体验三个维度。

---

## 二、需求优化列表

### 需求 1：审核不通过增加原因记录

#### 2.1.1 问题描述

当运营人员执行"审核不通过"操作时，系统仅记录状态变更（`hq_received` → `review_rejected`），未记录**不通过的具体原因**。

#### 2.1.2 当前实现

```
状态变更日志：
- archive_id: xxx
- status_field: status
- previous_value: hq_received
- new_value: review_rejected
- action: review_reject
- 无原因字段
```

#### 2.1.3 业务影响

| 影响对象 | 具体影响 |
|----------|----------|
| 分支机构 | 收到退回的合同后，不清楚问题所在，无法针对性修改 |
| 运营人员 | 多次打回同一档案时，无法快速回顾之前的问题点 |
| 管理层 | 无法统计分析常见的审核不通过原因，难以优化前置流程 |

#### 2.1.4 优化方案

1. **数据库变更**：`status_change_logs` 表增加 `remark` 字段（TEXT 类型，可为空）
2. **接口变更**：状态流转接口增加可选参数 `remark`
3. **业务规则**：当 `action = review_reject` 时，`remark` 为必填项
4. **前端变更**：审核不通过操作弹出原因填写框，支持常用原因快捷选择

#### 2.1.5 优先级

**高** - 直接影响业务协作效率

### 需求 2：寄送操作增加快递单号

#### 2.2.1 问题描述

分支机构执行"确认寄出"（`confirm_shipment`）和运营人员执行"确认回寄"（`confirm_shipped_back`）时，未记录**快递单号**信息。

#### 2.2.2 当前实现

仅记录状态变更，无物流追踪信息。

#### 2.2.3 业务影响

| 影响对象 | 具体影响 |
|----------|----------|
| 所有角色 | 无法追踪快递在途状态 |
| 运营人员 | 长时间未收到时无法查询快递状态 |
| 财务/审计 | 出现丢件纠纷时缺少凭据 |

#### 2.2.4 优化方案

**方案 A（推荐）**：利用现有 `remark` 字段记录
- 寄出操作时 `remark` 记录快递单号
- 格式约定：`快递公司:单号`，如 `顺丰:SF1234567890`

**方案 B**：新增专用字段
- `archive_records` 表增加 `shipping_tracking_no`（寄出单号）和 `return_tracking_no`（回寄单号）字段

#### 2.2.5 优先级

**中** - 提升物流追踪能力

### 需求 3：完善档案编辑权限控制

#### 2.3.1 问题描述

当前仅限制 `completed`（完全完结）状态的档案不可编辑，其他状态均可修改基本信息，存在数据一致性风险。

#### 2.3.2 当前实现

```typescript
// 仅此一处限制
if (record.status === 'completed') {
  return '该记录已完全完结，不可修改';
}
```

#### 2.3.3 业务风险

| 状态 | 风险场景 |
|------|----------|
| `in_transit` | 档案在途中被修改营业部，导致收件混乱 |
| `hq_received` | 审核中被修改客户姓名，影响审核判断一致性 |
| `return_in_transit` | 回寄中被修改，导致分支收到的与系统记录不符 |

#### 2.3.4 优化方案

建立分状态编辑规则：

| 状态 | 可编辑字段 | 说明 |
|------|------------|------|
| `pending_shipment` | 全部基本信息 | 尚未进入流程，允许完整修改 |
| `in_transit` | 无 | 在途中禁止修改 |
| `hq_received` | 无 | 审核中禁止修改 |
| `review_passed` / `review_rejected` | 无 | 审核结果已产生，禁止修改 |
| `pending_return` | 无 | 待回寄禁止修改 |
| `return_in_transit` | 无 | 在途中禁止修改 |
| `branch_received` | 无 | 等待流程完结，禁止修改 |
| `completed` | 无 | 已完结，禁止修改 |
| `null`（电子版） | 无 | 创建即完结，禁止修改 |

#### 2.3.5 优先级

**中** - 保障数据一致性

### 需求 4：记录档案打回次数

#### 2.4.1 问题描述

档案审核不通过后回寄，分支修改后重新寄出，系统自动回退到 `pending_shipment` 状态。但此过程没有记录**这是第几轮审核**，无法追溯历史。

#### 2.4.2 当前实现

```typescript
// 审核不通过路径：自动回退到 pending_shipment
result.sideEffects = [{
  statusField: 'status',
  previousValue: 'branch_received',
  newValue: 'pending_shipment',  // 无轮次信息
}];
```

#### 2.4.3 业务影响

| 影响对象 | 具体影响 |
|----------|----------|
| 运营人员 | 无法识别"问题档案"（多次被打回的档案） |
| 管理层 | 无法统计审核通过率、平均审核轮次等指标 |
| 分支机构 | 不清楚自己的档案质量情况 |

#### 2.4.4 优化方案

1. **数据库变更**：`archive_records` 表增加 `revision_count` 字段（INTEGER，默认 0）
2. **业务规则**：每次从 `branch_received` 回退到 `pending_shipment` 时，`revision_count += 1`
3. **查询支持**：支持按打回次数筛选档案
4. **可视化**：列表中显示打回次数，多次打回的档案高亮标识

#### 2.4.5 优先级

**中** - 支撑业务分析

### 需求 5：优化批量操作结果反馈

#### 2.5.1 问题描述

批量状态流转操作采用"逐条执行"模式，部分成功部分失败时，用户可能忽视失败记录。

#### 2.5.2 当前实现

```typescript
// 返回结果
{
  successCount: 7,
  failureCount: 3,
  results: [...]  // 包含每条记录的成功/失败详情
}
```

前端仅显示：`成功 7 条，失败 3 条`

#### 2.5.3 业务风险

- 用户可能以为选中的所有记录都已处理完成
- 失败的记录可能被遗忘，长期停滞在某个状态

#### 2.5.4 优化方案

**前端优化**：
1. 全部成功时：显示成功提示
2. 部分失败时：弹出**失败详情对话框**，列出失败记录及原因
3. 提供"仅查看失败记录"的快捷筛选

**后端优化（可选）**：
1. 提供"预校验"接口 `POST /api/archives/batch-transition/preview`
2. 先返回哪些记录可以执行、哪些不满足条件
3. 用户确认后再执行实际操作

#### 2.5.5 优先级

**高** - 避免业务遗漏

### 需求 6：归档与主流程状态协调提示

#### 2.6.1 问题描述

当分支确认收到回寄时，如果综合部归档尚未完成（`archive_status` 为 `pending_transfer` 或 `pending_archive`），主状态保持 `branch_received`，用户不清楚当前进度和下一步操作。

#### 2.6.2 当前实现

```typescript
// archive_status 为 pending_transfer 或 pending_archive 时
// status 保持 branch_received，等待归档完成后自动完结
```

#### 2.6.3 业务困惑

| 角色 | 困惑点 |
|------|--------|
| 分支机构 | 看到 `branch_received` 状态，不知道还需要等什么 |
| 运营人员 | 不清楚该档案是否还需要自己处理 |

#### 2.6.4 优化方案

**前端优化**：
1. 当 `status = branch_received` 且 `archive_status != archived` 时，显示特殊标签："等待综合部归档"
2. 增加状态说明提示："分支已收到回寄，归档流程进行中，完成后将自动完结"

**流程优化（可选）**：
1. 考虑主流程与归档流程完全独立
2. 主流程完结不依赖归档状态
3. 归档流程单独展示和追踪

#### 2.6.5 优先级

**中** - 提升用户体验

### 需求 7：支持导入更新已有记录

#### 2.7.1 问题描述

当前 Excel 导入功能仅支持新增，资金账号已存在时直接报错，不支持批量更新已有记录。

#### 2.7.2 当前实现

```typescript
const existing = this.archiveRepo.findByFundAccount(fundAccount);
if (existing) {
  errors.push({ row: rowNumber, reason: `资金账号 ${fundAccount} 已存在` });
  continue;
}
```

#### 2.7.3 业务限制

- 存量数据批量修正只能逐条编辑
- 迁移旧系统数据时，重复导入报错

#### 2.7.4 优化方案

1. **导入模式选择**：
   - 新增模式（默认）：资金账号存在则报错
   - 更新模式：资金账号存在则覆盖更新
   - 智能模式：存在则更新，不存在则新增

2. **更新限制**：
   - 仅允许更新 `pending_shipment` 状态的记录
   - 已进入流程的记录不可通过导入更新

3. **导入预览**：
   - 解析文件后先显示预览
   - 标识新增、更新、跳过的记录数
   - 用户确认后执行实际导入

#### 2.7.5 优先级

**低** - 提升批量操作能力

### 需求 8：增加档案作废机制

#### 2.8.1 问题描述

档案一旦创建，只能按流程向前推进，无法撤销或作废，录入错误的档案会永久保留。

#### 2.8.2 业务场景

| 场景 | 当前处理方式 |
|------|--------------|
| 录入错误（如资金账号填错） | 无法删除，只能备注说明 |
| 客户取消开户 | 合同作废无法体现在系统中 |
| 重复录入 | 第二条报"资金账号已存在"，第一条无法删除 |

#### 2.8.3 优化方案

1. **新增状态**：`voided`（已作废）
2. **操作权限**：仅运营人员可执行作废操作
3. **状态限制**：仅 `pending_shipment` 状态可作废（未进入流程）
4. **作废原因**：必填，记录作废原因
5. **数据处理**：作废记录保留但不参与业务统计，资金账号释放可重新使用

#### 2.8.4 优先级

**低** - 完善异常处理

---

### 需求 9：开户日期格式与合理性校验

#### 2.9.1 问题描述

导入和创建档案时，开户日期仅做字符串处理，未校验格式和合理性。

#### 2.9.2 当前实现

```typescript
openDate: String(row['开户日期']).trim(),  // 仅转字符串
```

#### 2.9.3 潜在问题

- 日期格式不统一（2024-01-01 / 2024/01/01 / 01-01-2024）
- 可能录入未来日期
- 可能录入明显不合理的历史日期（如 1900-01-01）

#### 2.9.4 优化方案

1. **格式校验**：统一为 `YYYY-MM-DD` 格式
2. **合理性校验**：
   - 不能是未来日期
   - 不能早于合理的业务起始日期（如 2000-01-01）
3. **格式转换**：导入时自动识别常见格式并转换

#### 2.9.5 优先级

**低** - 数据质量保障

---

### 需求 10：电子版合同增加来源标识

#### 2.10.1 问题描述

电子版合同创建即完结，但无法区分是"新开户"还是"存量数据补录"或"系统迁移"。

#### 2.10.2 业务背景

- 新开户的电子合同：正常业务流程
- 存量补录：历史数据补充到系统
- 系统迁移：从旧系统导入的数据

#### 2.10.3 优化方案

1. **数据库变更**：`archive_records` 表增加 `source_type` 字段
2. **枚举值**：
   - `new_account`：新开户
   - `backfill`：存量补录
   - `migration`：系统迁移
   - `other`：其他
3. **默认值**：新建和导入默认为 `new_account`
4. **统计支持**：支持按来源类型筛选和统计

#### 2.10.4 优先级

**低** - 数据分类完善
